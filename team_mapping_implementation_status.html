球队映射工具链实现状态

当前项目已经实现跨数据源球队映射候选生成、比赛证据聚合、我们自己的球队 proposal 生成，以及 DeepSeek/OpenAI-compatible LLM 复核。 所有脚本默认只读远程 MySQL，只写本地 outputs/ 文件，不会修改远程数据库。

thesports sr ls event evidence our_team proposal LLM verification

LLM 做了什么

LLM 不直接访问数据库，也不自由写 SQL。程序先把 proposal、source team 和事件证据整理成结构化 JSON， LLM 只负责判断这些 source team 是否代表同一支真实球队。 

{
  "same_team": true,
  "confidence": 0.95,
  "recommended_status": "llm_verified",
  "risk_flags": [],
  "reason": "Names and event evidence match with no youth/women/reserve conflict."
}

LLM verified 是复核通过，不等于正式 confirmed。正式 confirmed 仍建议依赖多场、多源、anchor 扩散验证。

准确度指标

当前没有把某一个分数直接当作最终准确率。球队映射准确度由多项指标共同判断： conflict_count、source_count、evidence_count、 team_confidence、avg_event_score 和 LLM 复核结果。

指标 含义 怎么计算/判断 使用建议
conflict_count 当前 proposal 中发现的明显冲突数量。 来自 evidence 中的冲突信号，例如比分明确不一致、主客方向冲突、同一 proposal 混入同源多个 team_id、 U19/U21/Women/Reserve 等层级不一致。 最关键指标。conflict_count = 0 才能进入高置信候选； conflict_count > 0 必须人工或更多证据复核。
source_count 有多少个数据源支持同一个球队 proposal。 统计 proposal members 里不同 source 的数量，例如 sr + ls + thesports 为 3。 source_count = 3/4 明显强于 source_count = 2。 后续接入 bc 后，四源支持的 proposal 应优先自动确认。
evidence_count 有多少条比赛证据支持这个球队映射。 每条 evidence 通常来自一次跨源同场比赛匹配，并由该比赛的主客队关系反推出球队 pair。 evidence_count = 1 是单场证据，只适合作为 seed candidate。 evidence_count >= 3 且无冲突，更适合进入 high confidence。
avg_event_score 支持该 proposal 的比赛候选平均分。 单场 event_score 综合时间差、队名相似度、主客方向、比分一致性、赛事名相似度。 proposal 级别取这些 event score 的平均值。 它是比赛匹配质量，不是球队最终准确率。 avg_event_score = 1.0 但 evidence_count = 1 时仍不能直接 confirmed。
team_confidence 当前脚本聚合出的球队 proposal 综合分。 初版公式： avg_event_score * 0.50 + source_score * 0.30 + evidence_score * 0.20 - conflict_penalty。 其中 source_score 来自 source 数量，evidence_score 来自 evidence 数量。 用于排序和筛选优先级，不等于真实准确率。 当前更适合判断“哪个 proposal 值得优先审核”。
llm_status LLM 对 proposal 的复核状态。 verify_team_proposals.py 把 members 和 evidence 发给 LLM， LLM 返回 llm_verified、needs_review 或 reject。 llm_verified 是二次审核通过，但不是最终 confirmed。 如果 LLM 给出 risk_flags，应继续查更多比赛证据。
llm_confidence LLM 自己给出的判断置信度。 由 LLM 根据名称、语言、球队层级、比赛证据和风险点输出。 辅助指标。不能单独用来确认映射，尤其不能替代多场、多源证据。
risk_flags LLM 或规则发现的风险标签。 常见标签包括 youth_level_mismatch、women_team_risk、 reserve_team_risk、abbreviation_only、insufficient_event_evidence。 只要存在风险标签，就不建议自动 confirmed。 后续可以触发更多数据库查询或人工复核。

判断规则

可作为 Seed Candidate

  • conflict_count = 0
  • avg_event_score >= 0.95
  • llm_status = llm_verified
  • 通常仍可能只有 source_count = 2、evidence_count = 1

高置信候选

  • conflict_count = 0
  • source_count >= 3 或 evidence_count >= 3
  • team_confidence >= 0.85
  • llm_status = llm_verified 且无 risk_flags

需要更多证据

  • source_count = 2 且 evidence_count = 1
  • 有缩写、翻译、青年队、女队、预备队风险。
  • LLM 返回 needs_review 或存在 risk_flags。
  • 应通过 anchor propagation 查询更多对战比赛。

不能自动确认

  • conflict_count > 0
  • 同一个 proposal 混入同源多个互斥 team_id。
  • U19/U21/Women/Reserve 与 senior team 无法解释地混用。
  • LLM 返回 reject。

准确度判断公式

当前脚本里的 team_confidence 是初版排序分，不是最终业务准确率：

team_confidence =
  avg_event_score * 0.50
  + source_score * 0.30
  + evidence_score * 0.20
  - conflict_penalty

source_score = min(source_count, 4) / 4
evidence_score = min(evidence_count, 6) / 6
conflict_penalty = min(conflict_count * 0.18, 0.54)

后续 anchor propagation 完成后，应把 anchor_team_count、opponent_consistency、 matched_event_count、source_event_count 加入公式，让它更接近真实业务准确度。

下一步

Anchor Propagation

  1. 从 LLM verified 的高质量候选中挑 seed confirmed team。
  2. 查询这些球队在各源里的更多比赛。
  3. 用已确认球队作为 anchor，对齐同一场比赛。
  4. 从同场比赛反推出对手球队映射。
  5. 累计多场、多源 evidence，形成真正的 team_confidence。
  6. 达到阈值后再进入正式 confirmed 映射。

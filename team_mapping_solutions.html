球队名称与球队 ID 统一映射方案

目标是把 thesports、sr、ls 以及后续 bc 的球队名称和源方球队 ID，映射到我们自己的统一球队体系。 这个问题属于跨数据源实体对齐，向量数据库可以用于候选召回，但最终真值应该落在可审计的关系型主数据表中。

核心原则

推荐原则

  • 我们自己维护一套 our_team_id，不要直接把某个数据源的 ID 当最终主键。
  • MySQL 保存最终映射关系，保证可查询、可回滚、可审计。
  • 规则、模糊匹配、向量库、LLM 都只作为候选推荐和辅助判断。
  • 高置信度自动确认，中低置信度进入人工审核。

主要风险

  • 同名球队可能分属不同国家、联赛、性别、年龄组或预备队。
  • 简称、翻译、本地语言、大小写、标点、FC/SC/CF 后缀都会造成名称差异。
  • 球队改名、迁移、合并会让历史数据和当前数据产生冲突。
  • 单纯靠字符串或向量相似度全自动匹配，容易产生难以发现的错误映射。

三个方案

方案一

MySQL 规则匹配 + 人工确认

适合第一版快速落地，数据规模可控，团队希望先把主流程跑通。

  • 抽取各源球队 ID、球队名、运动类型、联赛、国家。
  • 做名称标准化：小写、去标点、去常见后缀、统一空格。
  • 用精确匹配、token 匹配、Levenshtein/Jaro-Winkler 生成候选。
  • 人工确认后写入统一映射表和别名表。
方案二

MySQL 主数据 + 向量库候选召回

推荐作为中长期主方案，兼顾召回能力、可控性和后续扩展。

  • MySQL 存最终 our_team、source_team、team_mapping。
  • 向量库只存球队名称和上下文 embedding。
  • 新球队进入时，先用向量 topK 召回候选。
  • 结合规则分、向量分、运动类型、国家、联赛进行综合评分。
方案三

LLM 辅助疑难匹配

适合处理跨语言、历史改名、同名球队等低置信度 case。

  • 规则和向量先给候选 top 5，不让 LLM 自由搜索。
  • LLM 判断是否同一球队，并输出理由和置信度。
  • 只处理低置信度或冲突样本，控制成本。
  • 最终仍需写入可审核的映射表。

推荐落地顺序

  1. 先从 thesports、sr、ls 抽取唯一球队，形成 source_team。
  2. 建立 our_team、team_mapping、team_alias 基础表。
  3. 先跑规则匹配，产出第一批自动确认和待审核候选。
  4. 人工确认第一批映射，把确认结果沉淀为 alias。
  5. 加入向量召回，提升简称、多语言和不规则名称的召回率。
  6. bc 数据源接入时，只新增 source 抽取逻辑，不改变主模型。
  7. 低置信度和冲突样本再引入 LLM 辅助判断。
